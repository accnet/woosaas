## 4. What the Product Claims

From `README.md`, Woosaas positions itself as:

- analytics for WooCommerce store owners
- pageview and ecommerce event tracking
- attribution tracking
- revenue analytics
- funnel analysis
- realtime dashboard
- bot detection

This is a strong and coherent MVP story.

From `plan.md`, the MVP is more explicit:

- plugin installable on WooCommerce
- tracking for pageview, product view, add to cart, checkout, purchase

## 6. Main Gaps Between Docs and Code

### Gap 1: Setup docs understate the real ingest path

`docs/local-setup.md` tells the reader to run:

- infrastructure
- backend
- dashboard

But the actual ingest pipeline requires the worker process as well.

Why this matters:

- events are queued into Redis first
- the worker is responsible for writing them into ClickHouse
- without the worker, tracking may appear to succeed while reports remain empty or stale

This is the highest-priority documentation mismatch in the repository.

### Gap 2: The dashboard surface is smaller than the promised MVP

`plan.md` and `ANALYSIS.md` describe dashboard pages for:

- trend
- pages
- products
- onboarding

The backend already has query support for several of these areas, especially pages and products, but the dashboard does not expose them yet.

This creates a visible mismatch between the intended MVP and the current user-facing product.

### Gap 3: The repo contains future features that are not productized

There are meaningful stubs or service packages for:

- customer 360
- export
- helpdesk/support
- teams/permissions
- bot scoring beyond simple user-agent checks

Relevant packages:

- `api/internal/customer360/`
- `api/internal/export/`
- `api/internal/helpdesk/`
- `api/internal/teams/`
- `api/internal/bot/`

These are useful assets, but they are not yet part of the visible MVP.

### Gap 4: Some docs describe a more idealized architecture than the repo implements

Files like `api.md`, `analytics.md`, `data.md`, and `sql.md` are valuable, but they:

- use alternate event names in some places
- describe richer tracking or reporting than the current UI offers
- assume more mature documentation and folder structure than the repository currently has

These files should be read as design notes, not as operational truth.

### Gap 5: Bot detection is now usable, but the broader bot strategy remains partial

The codebase now supports bot reporting, but the broader strategy described in `bot.md` is only partially implemented.

Examples:

- `navigator.webdriver` checks are described in docs, but not fully wired into the tracker
- advanced server-side scoring exists conceptually in `api/internal/bot/scorer.go`, but is not fully integrated into ingestion flow
- some concepts in the docs assume access to raw IP intelligence, while the stored event model uses hashed IP for privacy

This is not a flaw in the MVP, but it is a clear gap between design ambition and current implementation.

## 7. What Is Already Strong

Several parts of the project are ahead of what a typical MVP would have:

- Redis Stream-based ingestion instead of direct single-row writes
- worker-based batch persistence to ClickHouse
- clean separation between business data and analytics data
- plugin test checklist
- real bot report instead of a placeholder
- reusable dashboard typing and shared utilities

This means the foundation is stronger than the UI suggests.

## 9. Recommended Next Actions

### Priority 0: repair operational docs

Update `README.md` and `docs/local-setup.md` so they clearly state:

- the worker is required for the full ingest path
- the worker should be started during local development
- tracking validation should include checking that events reach ClickHouse, not just the API

### Priority 1: close the MVP dashboard gap

Implement the dashboard pages already promised in the planning docs:

- trend
- pages
- products
- onboarding or tracking verification view

This is the most direct way to align the product with its own documentation.

### Priority 2: classify markdown files by status

Either by naming, headers, or folder organization, the repo should distinguish:

- current docs
- roadmap docs
- idea notes

Without that separation, contributors can easily implement against stale or speculative guidance.

### Priority 3: defer phase-two ideas until MVP surface is complete

Features like:

- helpdesk
- customer 360
- export center
- advanced bot scoring

should remain secondary until the documented MVP pages are complete and stable.

---

## 10. Final Assessment

Woosaas is not a vague prototype.

It already has:

- a credible ingestion architecture
- a real plugin path
- a real analytics backend
- a partially real dashboard

The main weakness is not the backend foundation.
The main weakness is alignment:

- docs say more than the UI currently delivers
- setup docs omit an important runtime dependency
- idea notes and implementation truth are stored side by side without clear labels

The highest-leverage path forward is not to expand scope.
It is to finish the documented MVP surface and tighten documentation around the architecture that already exists.
